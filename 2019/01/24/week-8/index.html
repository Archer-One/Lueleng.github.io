编译debug用glibc

64 bit

cd glibc-2.23 && mkdir build && cd build
CFLAGS="-g -g3 -ggdb -gdwarf-4 -Og"\
    CXXFLAGS="-g -g3 -ggdb -gdwarf-4 -Og"\
    ../configure --prefix=/path/to/install

32 bit

cd glibc-2.23 && mkdir build && build32
CC="gcc -m32" CXX="g++ -m32"\
    CFLAGS="-g -g3 -ggdb -gdwarf-4 -Og"\
    CXXFLAGS="-g -g3 -ggdb -gdwarf-4 -Og"\
    ../configure --prefix=/path/to/install --host=i686-linux-gnu

How DynELF Works

Pwntools:DynELF

  • 在libc版本未知时，透过information leakage寻找symbol

Lazy Symbol Binding

  • 程序启动时，外部函数的位置还未知
    • 例如定义在libc里的函数，gets，printf，….
    • 只有dynamically linked ELF才需要解析外部函数
      static ELF则直接包含了所有函数实作
  • 函数第一次被呼叫时，通过dynamic resolver来计算函数位置，并在GOT上做好relocation

Patch GDB for Debug Resolver

  • 原本gdb在step时会跳过dynamic resolver
  • Patch掉这段if
    #gdb-7.11/gdb/infrun.c:
    if(execution_direction != EXEC_RECERSE)

结构

ELF 结构

  • 档案结构/执行时记忆体结构 （readelf -aW a | less）
    Section & Segment
  • Section:档案中存放code或data的一块连续区块
    • 例：.text,.data,.bss,.got
    • 印出section内容：objdump -j .got.plt -s ./a
  • Segment:可以包含多个section，记忆体中的连续区块
  • Program header table描述section和segment的对应关系，不见得所有section都会被对应到，故section不一定会出现在记忆体中
ELF Header
  • 一些格式设定值，包括是x86还是x64
  • Program header table和section header table 位置
  • 执行时会放在base，检查开头magic是以这种确认是否为base address的方法
Section Header
  • Section header table 为ELF64_Shdr的阵列
  • 指出每个section的位置
  • 跳过，反正执行时看不到这个
Program Header
  • Program header table 为ELF64_Phdr的阵列
  • 指定档案中内容和在记忆体中的位置，某个范围(segment，可以包含多个section)要被载入到哪个地址
.dynamic Section
  • 解析symbol时重要的section
  • 虽然没有section header table，但可以从program header table 找到它
    • 找出p_type值为PT_DYNAMIC的program header
    • Base加上p_offset即为.dynamic
  • 是Elf64_Dyn
Dynamic Entry
  • 用d_val还是的d_ptr取决于d_tag(为DT_xxx)
  • .dynamic中d_tag为DT_SYMTAB的entry，d_prt指向.dynsym section
  • 为Elf64_Sym阵列
    • st_name指向symbol name字串
.dynstr Section
  • d_tag为DT_STRTAB
  • 为.dynsym中st_name对应的string table
    • name=(char *)(.dynstr + st_name)
.rel.plt(.rela.plt)
  • d_tag为DT_JMPREL,结构为Elf64_Rel(Elf64_Rela)
  • r_offset为要做relocation的位置，即GOT entry
  • r_info中包含symbol index
    • ELF64_R_SYM取高32bit;ELF32_R_SYM取高24bit
    • Symbol index为.dynsym中的index
解析GOT上的函数
  • GOT entry原本的值是.plt entry 中的第二条指令
    • push reloc_arg(.rel.plt中的offset)
    • 跳到.plt中的第一行(PLT0)
  • PLTO
    • push(GOT1),为指向link_map结构的指标
    • jmp(GOT2)，跳到dynamic resolver开始寻找symbol
Dynamic Resolver
  • 需要link_map和reloc_arg作为参数
  • struct link_map包括所有已载入的ELF资讯
  • 取得symbol nam，在library里找到后再填回GOT
link_map结构
  • l_next-串接所有已载入library
  • l_name-library name
  • l_addr-library base address
  • l_info[x]-指向.dynamic中，d_tag=x的栏位
    • 可以拿到library中各个section
Symbol Resolution
  • _dl_runtime_resolve->_dl_fixup(link_map,reloc_arg)
  • 理论上是可以直接扫整个.dynsym，检查sy_name看看是不是等于要找的symbol，但太花时间
  • Resolver透过hash table来找
    • GNU hash
查找Hash Table
  • int b = l_gnu_buckets[hash % l_nbuckets]
  • i = b 开始，依序检查
    ((l_gnu_chain_zero[i]^hash)>>1)==0
    直到(l_gnu_chain_zero[i]&1)!=1为止
    • 若相等，sym=.dynsym[i],即低i个Elf64_Sym
    • 再检查sym->st_value+l_addr，即为该symbol在library里的位置
