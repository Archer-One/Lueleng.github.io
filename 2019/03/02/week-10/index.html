Heap Exploitation

Glibc Heap Overview

  • 负责维护动态配置记忆体的资料结构成为heap
  • Libc里比较常用的部分是malloc，free，realloc
    • C++的new，delete，底下的实作仍是上述的几个
  • 配置新记忆体、释放并回收不需要的部分，在配置记忆体的时候尽量避免[碎片化]

    Heap相关的漏洞利用

  • Use after free
  • Double free
  • Heap overflow

    Glibc Heap是做概况

  • 要做到记忆体的管理，需要知道

    • 有哪些位置的记忆体是可以被分配的
    • 有哪些是因为free掉而可以回收的
    • 有哪些位置是使用中则不需要记录，使用它们的人应该记住这些指标
    • 整个Heap的资料记录在一个struct malloc_state中，成为main_arena
    • malloc分配的记忆体成为chunk，会比要求的大小要大一点，因为需要记录一些维护heap用的额外资料
    • Arena跟h eap分配的记忆体分开存放，heap overflow没办法直接去盖掉它的内容
    • 回收的chunk用linked list记录，称为bin
    • main_arena中有很多bin，每个bin里存储的chunk size不同，目的是让malloc时可以尽快找到最适合大小的chunk

    • 回收的chunk会依据size来决定应该放进哪个linked list(bin)中
      main_arena{

      bin[0] (size=16) -> chunk1 -> chunk5

bin[1] (size=32) -> chunk2 -> chunk3 ->chunk4

bin[2] (size=48) ->

      }

    • malloc时，会先从bin里面找出可以使用的chunk，如果找不到才会真的分配新的记忆体给程序使用。分配时可以去找到足够大的chunk只切出需要的部分，剩下的部分形成新的chunk(last_remainder)
    • 找不到可用空间时会从top chunk分配，top chunk是一个很大的chunk，代表可使用但未分配的记忆体，malloc分配时会从里面切一小块下来，剩下来的部分重新设为top
      main_arena{
      mchunkptr top -> top_chunk;
mchunkptr last_remainder;
      }

      Chunk

  • 存放chunk metadata的chunk结构(header)
    struct malloc_chunk{
    size_t prev_size;
size_t size;
malloc_chunk* fd;
malloc_chunk* bk;
malloc_chunk* fd_nextsize;
malloc_chunk* bk_nextsize;
    }
  • (64bit): mem = malloc(size)
    -> chunk = mem -16; chunksize = (size+8)#16
    • 实际chunk位置是malloc得到的位置-16
    • chunksize是size +8后向上对齐至16的倍数，因为要存放额外的资料

      Chunk Header 栏位

  • size:这个chunk在记忆体里的大小，非malloc size
  • fd，bk:指向bin里的前一、后一个chunk
    • 一般来说bin是double linked list
  • prev_size:前一个chunk size，维护heap时可以得知前一个chunk的位置
  • 概略来说，每个chunk可以看到四个相邻的chunk
    • chunk + size:记忆体空间里的下一个chunk
    • chunk - prev_size:记忆体空间里的前一个chunk
    • fd,bk:Double linked list(bin),在记忆体中不一定是在前或后

Exploit: Use After Free

  • 跟chunk，bin等heap的内内部操作
  • 让程序有两个指标指向同一个记忆体
    • 一块是structure，另一块是用作data buffer
    • 利用对buffer的读写，修改或泄露structure的内容

利用Chunk的回收特性

  • 所谓的use after free，是指free(=p)掉后还继续使用它
    • 例：linked list remove掉物件后，忘了把指向它的指针清NULL，导致以为此物件还存在list中
  • 重新malloc(=q)一样的大小，会拿到曾经free掉的chunk，此时就存在两个指针p，q指向同一快记忆体，使用这两个指针的操作会混在一起
    • 例：其中一个是C++物件，有个vtable指针用来找出实际的function。如果另一个是可写入的data buffer，就可以改掉function pointer
  • 只要实际的chunk size相同，malloc要求的大小不同也没关系
  • 会拿出哪个chunk，取决于heap里bin的运作方式，不一定是最近free的那一个

Fastbin

  • Chunk size<= get_max_fast()的chunk，会被放在一系列称为fastbin的bin里
    • 64bit是128bytes，32bit是64bytes
    • global_max_fast一开始是0
  • Fastbin是single linked list，只使用fd，以NULL结尾
  • Chunk size从32开始，共7个可用的fastbin
  • free是不取消下个chunk的prev_inuse bit,因为fastbin chunk不会和其它chunk合并
  • malloc、free操作时glibc会有一些检查，确认heapp metadata是否正确，避免一些可能的攻击方式
  • 为了执行效率，fastbin里的检查比其它类型的bin少很多

Fastbin Corruption

  • 让fastbin linked list指向任意位置，之后malloc时就会将该位置作为chunk拿出来
  • Freed(not inuse)chunk才会在bin里，修改它的fd才会造成corruption
    • Double free
    • Overflow

      Fastbin Sanity Check

  • malloc从bin里取出时，要从正确的bin里拿出来，及chunk size要正确
  • free时，nextchunk的size要对
  • free时会检查bin里第一个chunk是不是跟现在这个要free的这一个相同(fasttop)

Overflow Freed chunk

  • 如果有heap overflow，可以覆盖下个chunk的fd指标
  • 先free掉下一个chunk再进行overflow
  • Overflow时要注意下个chunk的size是否正确

fastbin2.c

Free Arbitrary Address（House of Spirit）

  • 其它overflow可以改掉一个原本由malloc得到的指标p，free(p)时会使得被改掉的指标进入fastbin，下次malloc时就会拿到

Unlink Exploitation

freed chunks

  • 非fastbin chunk在free掉是，会跟前后的freed chunks合并

unlink() Macro

  • unlink()是用来把chunk移出bin用的，从double linked-list移除

合并freed chunks

  • 非fastbin chunk在free掉时，会跟前后的freed chunks合并

利用unlink()–old

  • 如果有overflow可以覆盖到某个chunk q的prevsize，free q时传入unlink(p)的p可以控制
  • 慎选p使得chunk p的内容也可控
  • 利用FD->bk = BK 和 BK->fd = FD, 可以同时写入两个目标，例如：
    • FD = p->fd = free@got.plt - 0x18
    • BK = p->bk = shellcode

Sadly Truth

  • 实际的unnlink会检查double linked-list是不是合法
    • 指过去要能指回来，即p->fd->bk == p

Overwrite Heap Pointer

  • 实际上unlink的检查还是有办法绕过，而且可以得到一种很实用的利用方法
  • 需要有：
    • 一个指向heap内的指标
    • 存放该指标的位置一直(例如：该指标是全域变数)
    • 可以对该指标写入多次
      • FD = p->fd = &p - 0x18
      • BK = p->bk=&p - 0x10
      • FD->bk = BK
      • BK -> fd =FD
      • Result: p = &p - 0x18

mmap and arena

mmap chunks

  • malloc size 超过约0x21000时，会改用mmap直接要求新的记忆体
  • mmap得到的位之时连续的，得到的chunk会在上一次mmap之前，通常最后一次mmap会是tls段

伪造arena

  • 平常使用的arena是在libc内部的main_arena变数
  • malloc时其实会根据tls段上面的某个指标，决定要使用的arena
  • mmap chunk overflow 时可以盖掉arena指标
    • tls段上还有stack address ，stack guard canary
  • 伪造arena的fastbin部分，使得下次malloc时可以取得伪造的chunk
  • 这个利用方式需要任意大小的malloc，但不需要free
